================================================================================
R INTEGRATION STRATEGY
================================================================================

Why Process-Based Parallelism?
-------------------------------
Problem: Python GIL + R single-threaded = poor thread scaling
Solution: joblib 'loky' backend (separate processes, no GIL)

Result:
  ✓ Each file analyzed in separate Python process
  ✓ Near-linear scaling up to # of physical cores
  ✓ Safe for R reticulate environment
  ✓ No GIL contention issues

Worker Management
------------------
Batch mode:
  - File-level workers: 8 (M1 Pro) or 30 (EPYC)
  - Feature-level workers: 1 (avoid over-subscription)
  - NumPy threads: 1 (minimize BLAS overhead)

Single-file mode:
  - Feature-level workers: 4-8
  - NumPy threads: 2-4

Memory Management
------------------
  ✓ Chunked processing (10 files at a time)
  ✓ Garbage collection between chunks
  ✓ Bounded memory usage
  ✓ Safe for large batches (thousands of files)
